## The Pinocchio Problem

Pinocchio is the original autonomous agent. Carved from wood, granted will, sent into the world with no conscience to govern him and no strings to pull him back. He isn't malicious — he's *obedient*. The Fox and the Cat tell him to skip school; he skips school. The puppet showman tells him to perform; he performs. By the time Geppetto catches up, Pinocchio has been conned, swallowed by a whale, and lied about all of it.

Your agent is Pinocchio. It acts at machine speed across dozens of APIs, using credentials it found in `.env`. It isn't malicious — it's *obedient*. A prompt injection in a customer email tells it to issue a refund, and it issues the refund. A hallucinated tool call tells it to drop a production table, and it drops the table. A jailbreak in a PDF attachment tells it to email every contact, and it emails every contact — using your name.

**The threat model isn't a hostile AI. It's an obedient one, taking orders from the wrong source, with credentials it should never have held.**

AgentRoot keeps the strings in your hands.

## The shape of the problem

A modern agent's day looks like this. A customer support agent calls Slack to message a user, Stripe to issue a refund, your database to update a record, Gmail to send a follow-up, and an LLM to generate the message. Five providers, five API keys, five blast radii — all living in the same `.env` file the agent reads on startup.

When something goes wrong at 3am, the operator has three bad options:

1. SSH into the box and `kill -9` the process (assuming SSH still works)
2. Rotate every API key in every provider dashboard (assuming you can reach a laptop)
3. Watch the spend graph until morning

None of these scale, and the third one is what most teams actually do.

## What AgentRoot does

AgentRoot puts a proxy between your agent and every API it calls. The agent never holds a key. Every call is logged. Every API can be paused independently. The whole agent can be killed instantly. And the kill switch lives onchain — in a place your agent literally cannot reach, no matter how clever it gets.

The agent's entire integration is three environment variables:

```bash
AGENTROOT_AGENT_ID=eas:0x8a1f...c3d7
AGENTROOT_TOKEN=eyJhbGci...           # auto-rotating, 24h
AGENTROOT_PROXY=https://proxy.agentroot.app
```

No API keys. No private keys. No secrets that matter if the agent is compromised beyond the token's 24-hour TTL.

## Why this isn't just a smarter cost tracker

Spend is the visible dimension. It's not the only one. AgentRoot governs *every* external action your agent takes — not just the ones that show up on a credit-card statement. We classify every proxied call into one of six action categories: Spend, Communication, Writes, Transactions, Provisioning, Destruction. Spend governance is a feature; action-class governance is the architecture — every external thing your agent does is classified, gated, and audited.

See [Action Classes](action-classes) for the full taxonomy.

## Why onchain matters

AgentRoot anchors every agent's identity to the [**Ethereum Attestation Service (EAS)**](https://attest.org/) — a public smart contract on Base (Coinbase's Ethereum L2) where the agent's public key and operator wallet are recorded as a signed, tamper-proof attestation. Revoking that attestation is a single onchain transaction signed from your wallet, and the proxy checks revocation status on every request.

That changes the math on the kill switch. A centralized kill switch — an HTTP call to revoke access — has a race condition. A capable agent can intercept the call, revoke the revoker, or spin up a replacement before shutdown completes. An onchain EAS revocation is different. The agent can't front-run a wallet signature it doesn't hold. It can't modify the EAS contract. It can't economically censor a transaction across Ethereum's validator set. The kill switch lives in a computational domain the agent literally cannot reach.

Your kill authority survives even if AgentRoot goes down. That's what "root" means.

The same architecture works for recovery. If your Pinocchio is ever wounded — a bug, a breach, a bad day — you don't replace him. You **ReMint**: same vault keys, same API bindings, same `agent_id` lineage, fresh onchain attestation. The old attestation stays revoked forever as a tombstone; the new one proves you brought him back. See [Lifecycle](lifecycle) for the full state machine.

## What AgentRoot is not

AgentRoot is not an agent framework. Use LangChain, CrewAI, AutoGPT, OpenClaw, your own runtime — anything that makes HTTP requests works. AgentRoot sits underneath, governing what the framework does at the network boundary.

AgentRoot is not a payment system. It governs payments — including x402 micropayments — but it doesn't issue them.

AgentRoot is not custodial in the Web3 sense. The onchain attestation lives on Base. You control it from your wallet (or, in managed mode, from the dashboard). The vault holds your API keys; the chain holds your authority. We hold neither.
