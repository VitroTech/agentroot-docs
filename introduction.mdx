# Introduction

Your agent is one prompt injection away from emptying your Stripe account, deleting your production database, and posting an apology in #general — using credentials it found in `.env`. The agent never needed to be malicious. It just needed to follow instructions you didn't write.

This is the agentic security problem in one sentence. AgentRoot is the answer.

## The shape of the problem

Modern agents act at machine speed across dozens of APIs. A customer support agent calls Slack to message a user, Stripe to issue a refund, your database to update a record, Gmail to send a follow-up, and an LLM to generate the message. Five providers, five API keys, five blast radii — and they all live in the same `.env` file the agent reads on startup.

When something goes wrong at 3am, the operator has three bad options:

1. SSH into the box and `kill -9` the process (assuming SSH still works)
2. Rotate every API key in every provider dashboard (assuming you can reach a laptop)
3. Watch the spend graph until morning

None of these scale, and the third one is what most teams actually do.

## What AgentRoot does

AgentRoot puts a proxy between your agent and every API it calls. The agent never holds a key. Every call is logged. Every API can be paused independently. The whole agent can be killed instantly. And the kill switch lives onchain — in a place your agent literally cannot reach, no matter how clever it gets.

The agent's entire integration is three environment variables:

```bash
AGENTROOT_AGENT_ID=eas:0x8a1f...c3d7
AGENTROOT_TOKEN=eyJhbGci...           # auto-rotating, 24h
AGENTROOT_PROXY=https://proxy.agentroot.app
```

No API keys. No private keys. No secrets that matter if the agent is compromised beyond the token's 24-hour TTL.

## Why this isn't just a smarter cost tracker

Spend is the visible dimension. It's not the only one. AgentRoot governs *every* external action your agent takes — not just the ones that show up on a credit-card statement. We classify every proxied call into one of six action categories: Spend, Communication, Writes, Transactions, Provisioning, Destruction. Twenty-seven of our fifty-four launch services are action-class — exactly half. That's the thesis in data form: AgentRoot's differentiated value is governing what your agent *does*, not metering what it *thinks*.

See [Action Classes](action-classes) for the full taxonomy.

## Why onchain matters

A centralized kill switch — an HTTP call to revoke access — has a race condition. A capable agent can intercept the call, revoke the revoker, or spin up a replacement before shutdown completes. An onchain EAS revocation is different. The agent can't front-run a wallet signature it doesn't hold. It can't modify the EAS contract. It can't economically censor a transaction across Ethereum's validator set. The kill switch lives in a computational domain the agent literally cannot reach.

Your kill authority survives even if AgentRoot goes down. That's what "root" means.

## What AgentRoot is not

AgentRoot is not an agent framework. Use LangChain, CrewAI, AutoGPT, OpenClaw, your own runtime — anything that makes HTTP requests works. AgentRoot sits underneath, governing what the framework does at the network boundary.

AgentRoot is not a payment system. It governs payments — including x402 micropayments — but it doesn't issue them.

AgentRoot is not custodial in the Web3 sense. The onchain attestation lives on Base. You control it from your wallet (or, in managed mode, from the dashboard). The vault holds your API keys; the chain holds your authority. We hold neither.
