The dashboard at [ux.agentroot.app](https://ux.agentroot.app) is your control plane. Most operators only need four workflows: onboarding a new agent, daily monitoring, triaging a suspicious call, and going to sleep. This page covers each in order, then leaves the rest as reference.

For incident-response runbooks — the moments where you need a script, not a tour — see [Operator Playbooks](playbooks).

## Sign in

Two sign-in methods, both give full dashboard access:

- **Social login (Google or GitHub)** — fastest path. AgentRoot's managed attester (`agentroot.eth`) signs onchain attestations on your behalf. You get every dashboard feature; you don't get sovereign Tier 3 (your wallet revoking onchain, independent of AgentRoot).
- **SIWE (Sign-In with Ethereum)** — sovereign mode. Your wallet is your operator identity. Tier 3 [Kill](kill-switch) revocations sign from your wallet directly against the EAS contract on Base.

Migration from social to SIWE without re-registering agents is in development. If you might want sovereign control eventually, sign up with SIWE now.

## Onboarding a new agent

The first thing you'll do. Four clicks, one proxied call.

1. **New Agent.** Provide a name and optional description. The vault generates a P-256 keypair; the EAS attestation is signed (your wallet in sovereign mode, AgentRoot's attester in managed mode); the agent is live.
2. **Copy credentials.** Three values: `AGENTROOT_AGENT_ID`, `AGENTROOT_TOKEN`, `AGENTROOT_PROXY`. The "Copy All" button puts a ready-to-paste `.env` block on your clipboard. Drop them into your agent's environment.
3. **Bind an API key.** Click **Add API Key**. Pick a provider, paste the key into the secure iframe (it goes straight to the vault — the dashboard never sees plaintext), label it. The dashboard never displays the value again.
4. **Make your first call.** The wizard's last step shows a ready-to-run snippet in cURL, TypeScript, and Python. Run it. The activity stream shows the call within a second or two. You're live.

Full step-by-step is in [Quickstart](quickstart) (split by sign-in method).

## Daily operations

The activity stream is the dashboard's center of gravity. Every proxied call appears with: timestamp, agent, provider, [action class](action-classes), path, status, latency.

**Filter aggressively.** By agent for per-agent triage. By provider when an upstream is misbehaving. By action class when you want to see all Communication events across every agent (or all Destruction events — that filter should always return zero unless you've explicitly allowed something). By time window when reviewing a specific incident.

**Set per-class thresholds on every agent.** Auto-kill is the safety net you should never deploy without. The configuration lives at Agent → Auto-Kill Thresholds. Common defaults to start with:

- Spend: $X/day, $Y/month, hard cap $Z/call
- Communication: max N sends per recipient per day; off-hours pause
- Destruction: deny by default; explicit per-resource allowlist; HITL on the rest
- Provisioning: max M sub-agents per parent

The values matter less than the existence — anything that can grow unbounded should have a ceiling.

**Bulk operations across agents.** From the agents list, multi-select to disable several agents at once, rotate a key bound to multiple agents, or apply the same threshold pattern to a group. Bulk Disable is the right move when an upstream provider is having an incident — pause every agent that touches them in one click.

## Triage: investigating a suspicious call

You see something in the stream you didn't expect. Walk it down.

1. **Open the call.** Metadata only — request and response bodies are not stored — but everything else is there. Provider, path, method, status, latency, action class, timestamp.
2. **Look at the calls before it.** Same agent, same time window. The trigger usually reveals itself in the immediately preceding traffic — a user message processed, a webhook fired, a scheduled job that ran.
3. **Decide on scope.** One provider misbehaving for one agent → Disable the binding (Tier 1). Whole agent looks wrong → Disable the agent (Tier 2). Pattern repeats across agents → adjust thresholds globally.
4. **Document and re-arm.** Add a note to the agent. Adjust thresholds if needed. Re-enable. The audit log keeps the record either way.

Full version with context: [Playbooks → Investigating a suspicious call](playbooks#investigating-a-suspicious-call).

## Sleep: leaving for the weekend

Pre-flight before you stop checking the dashboard. Five items.

- **Auto-kill thresholds set on every agent** (see above)
- **Notification routing tested** — send yourself a test alert, confirm it lands on the device you'll have with you
- **Mobile dashboard works on your phone** — sign in, reach an agent's detail page, see the Disable button
- **For SIWE users — your wallet is reachable** on a mobile signer
- **No risky deploys after lunch on Friday**

If all five check, the architecture is doing its job. Close the laptop.

Full checklist with rationale: [Playbooks → Leaving for the weekend](playbooks#leaving-for-the-weekend).

## Reference

The rest of the dashboard, organized by surface.

### Agents

Each agent's detail page shows status (Active / Paused / Killed), the EAS attestation UID with a link to the onchain record, connected API keys with their bindings, the activity stream filtered to this agent, the session token TTL and last refresh, and the auto-kill threshold configuration. Buttons for Disable, Kill, and (on Killed agents) ReMint.

### Keys

The keys list shows every API key in the vault, with the providers they're bound to, the agents that use those bindings, and the last-used timestamp. Per-key actions: Disable (instant Tier 1 across every agent), Replace (new key swapped into the same bindings), Delete (vault ciphertext purged). The label and provider are visible; the key value is never displayed after ingestion.

### Activity

The activity surface is the unfiltered stream plus a query builder. Filters: agent, provider, action class, status code, time window. Export to CSV for offline analysis. Streams update in real time.
